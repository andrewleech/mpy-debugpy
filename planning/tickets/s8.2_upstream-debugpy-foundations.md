## Design constraints

- Upstream-first: commits must stand alone for micropython-lib review — no
  references to this repo's roadmap/journey in code comments or commit messages.
- The no-thread model is a hard constraint the reviewers know; `wait_for_client`
  must keep its poll-loop shape (same as `wait_for_continue`).
- Capability truth: probe at runtime, never trust names — this is the selling
  point of the probe commit; say so in the PR text.
- The readOnly hint must degrade gracefully on DAP clients that ignore
  presentationHint (it does — hint-only).

## Approach sketch

Decide the reconciliation order first (it determines s8.5's micropython-lib
branch form):

- (a) **Jos's layer first**: merge/land `andrewleech/micropython-lib#2` into
  `add-debugpy-support`, then rebase the foundations on top — matches the
  integration's actual ancestry, no ruff conflicts, but couples our fix to the
  enhancement layer's review.
- (b) **Foundations first** (recommended for the framing fix): cherry-pick
  `61c8905` back onto `5d491e0` (small, surgical — messaging.py predates the ruff
  sweep in relevant hunks; verify), push to `add-debugpy-support` so PR #1022
  itself gets the correctness fix ASAP; rebase `e359a95` later, after deciding on
  Jos's layer.

Either way, run this repo's `make test` harness against the resulting branch
before pushing (point the micropython-lib submodule at it locally).
