# mpremote+wrapper: Hash-verified incremental source sync with on-device manifest (fallback path)

Phase: EPIC-4 / STORY-4.2
Depends on: s4.1 (install/verify plumbing and the verify_hash decision), D2 (defines when this path is used at all)
Written: 2026-07-06 at HEAD 59fab8450a (micropython 18c76f7c56, micropython-lib 0d42fcd5bc; ampremote/micropython 5b459e2d57)

## Context

D2 (`../SPIKE-2.1-mount.md`) made `mpremote mount` the MAINLINE source-delivery path:
mounted source is always live, so no sync engine is needed there. This ticket is the
**fallback only**, for sessions where mount is unavailable or undesired: no-mount
network-attach sessions (device runs standalone after the serial control phase ends,
e.g. the serial line must be released), single-UART boards where the serial line will be
dropped mid-session, or a user-requested "copy, don't mount" mode. It replaces the old
wrapper's `cp -r src/ :/`-on-every-run with per-file hash diffing.

Constraining decisions: D2 (fallback status — do not let this grow into the default),
Q3 (single-UART boards ship network-only in v1 — they are this ticket's main customer).

## Scope

In scope: incremental push of the user's project source to the device, driven by an
on-device manifest `:/.mpdbg/manifest.json` mapping `device_path → sha256`; recording
the local→device path map for s4.3's pathMappings generation.
Out of scope: debugpy package install (s4.1); attach-time staleness warning (s4.3 —
this ticket produces the data it reads); any mount-mode behaviour (mount needs no sync).

## Files and anchors

- Transfer primitives: ampremote lineage `~/ampremote/micropython/tools/mpremote/
  mpremote/transport.py:292` `fs_writefile(..., verify_hash=True)` (device-side sha256
  during transfer, #18436) and `:194` `fs_readfile(..., verify_hash=True)`. This repo's
  mpremote submodule copy (`micropython/tools/mpremote/mpremote/transport.py:154`) lacks
  verify_hash — the s4.1 open question (register #18436 into this repo's integration)
  gates this ticket too.
- Fast writes: #18785 smart-encoding fs_writefile (ampremote branch
  `feature/smart-encoding-fs-writefile`) — use if present, don't require it.
- Old behaviour being replaced: the original wrapper's `upload_all.py` / `mpremote cp -r
  src/ :/` full-recopy (see ROADMAP "The current friction").

## Design constraints

- Manifest is on-device and updated only AFTER a verified transfer, so an interrupted
  sync self-heals on the next run.
- Local hashing must include every file the target imports, not just the entry module —
  default: all `*.py`/`*.mpy` under the configured source root(s) from the target model
  (s5.2's `mpdebug.toml`).
- The local→device path map is written to the host session state in a form s4.3 can turn
  into absolute DAP pathMappings — this is the fix for the old multi-root ambiguity.
- Deleted local files: remove from device (and manifest) rather than leaving stale
  modules importable — a stale-but-present module is exactly the failure this epic
  exists to kill.
- Serial-friendly: raw-REPL transport primitives only.

## Approach sketch

1. Read `:/.mpdbg/manifest.json` (missing → empty).
2. Hash local files under the source roots; diff against manifest.
3. Push changed/new files with `verify_hash=True` (+ #18785 encoding when available);
   delete device files absent locally; update manifest last.
4. Emit the sync record `{local_abs_path → device_path}` for s4.3.
5. Wire as the `sync` step of resolve→ensure→sync→launch→attach ONLY when the resolved
   target mode is no-mount (mount mode skips straight past it).

## Acceptance criteria and tests

- Unchanged tree → second sync performs zero file writes (assert via transport spy).
- One changed file → exactly that file is pushed and the manifest entry updates.
- Locally deleted file → removed on device and from the manifest.
- Simulated hash-verify failure → loud error, manifest NOT updated, next run retries.
- Interrupted sync (kill between write and manifest update) → next run re-pushes.
- Adversarial: device manifest hand-edited to lie (hash matches manifest but device file
  differs) — document that this is out of detection scope here (s4.3's attach-time spot
  check is the guard) or add a `--verify-all` re-read mode; either way the behaviour is
  explicit, not silent.
