## Files and anchors

- Branch creation pattern (`~/ampremote/CLAUDE.md`, "Adding a local-only branch"):
  branch from upstream master in the submodule
  (`git -C micropython checkout -b <branch> master`), push to the
  `andrewleech/micropython` fork.
- `~/ampremote/mbm.toml` â€” `[[submodules]]` path `micropython`, integration_branch
  `ampremote`; branches are `[[submodules.branches]]` entries with `name`,
  `pr_url`, optional `pr_number`, `title`. Local-only branches use a fork tree URL
  (pattern: `mpremote_raw_repl_retry` entry). **`ampremote_rename` must stay last
  in merge order** â€” insert the debug entry before it.
- `mbm add-pr` / manual-entry + rebuild flow: on conflict, resolve,
  `git merge --continue`, then fast-forward the integration branch
  (`git -C micropython branch -f ampremote <tip>`); rerere records resolutions.
- ampremote install model: `make install` in `~/ampremote` does
  `uv tool install --editable` of `micropython/tools/mpremote`, so once merged the
  command is live as `ampr debug` for hardware validation.
- This repo's side: the same branch becomes an entry in
  `/home/anl/mpy-debugpy/mbm.toml` under the micropython submodule (after
  `pdb_support` and the locals branch), so `mbm rebase` composes firmware debug
  support AND the mpremote command into one submodule checkout.

## Design constraints

- One branch, two consumers â€” never patch the command differently in the two
  integrations; fix on the branch and rebuild both.
- mpremote house style (per STORY-5.1 research): commands registered in the
  `_COMMANDS` dict in `micropython/tools/mpremote/mpremote/main.py`; handlers in
  `commands.py` take `(state, args)`; new-arg defaults preserve old behaviour.
- `ampremote_rename` stays last in ampremote's merge order.

## Approach sketch

1. Branch `mpremote_debug` (name it to match ampremote's `mpremote_*` convention)
   off upstream master on the fork; the EPIC-5 work lands there.
2. Register in `~/ampremote/mbm.toml` before `ampremote_rename`; rebuild the
   ampremote integration; `make install`; smoke `ampr debug --help`.
3. Add the entry to this repo's `mbm.toml`; `mbm rebase -s micropython`; verify
   `make firmware-unix` still builds (the branch only touches `tools/mpremote`,
   so firmware is unaffected â€” the value is one checkout carrying both).
4. Open the draft PR on `andrewleech/micropython` (base: the fork's master mirror)
   per policy; body references the mpy-debugpy planning docs.
