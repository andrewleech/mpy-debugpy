## Design constraints

- ampremote house style (from `~/ampremote/CLAUDE.md`, verbatim rules): inline byte
  literals with trailing intent comments (no control-byte constants); prefer extending
  the existing `read_until` / `exec` primitives over new abstractions; new parameters on
  existing methods default to previous behaviour; single-caller helpers get a leading
  underscore.
- Upstream-first (D1/EPIC-8): code must be upstreamable as-is — no `ampremote`-specific
  naming (`ampremote_rename` handles branding separately and stays last in mbm order).
- Endpoint truth comes from the device (`MPDBG-READY`), never from user config
  (design position; enforced by s5.4).
- The debugpy server is single-threaded/poll-driven — the command must not assume it
  can talk DAP before the client attach phase; the boot script blocks in
  `wait_for_client()` until `configurationDone`.

## Approach sketch

1. `argparse_debug()`: description "debug a MicroPython script with a DAP client";
   positional `target` (name from `mpdebug.toml`, resolved in s5.2 — until then accept
   `unix` or a serial connect string), optional positional `program`
   (`module[:method]`, default `target:main` matching the launcher's defaults),
   `--port` (int, default 0 = device-assigned), `--dap-log` (declared, wired in s6.3).
2. `do_debug(state, args)`: parse `module[:method]`; branch on target kind. For the
   serial stub: `state.ensure_raw_repl()`, `state.did_action()`, upload/exec the boot
   script with args via the `_do_execbuffer` shape, read output lines until
   `MPDBG-READY ` prefix, `json.loads` the remainder, print endpoint+caps. Unix branch
   raises "see s5.3" NotImplemented-style `CommandError` until s5.3 lands.
3. Keep the handshake-read helper private (`_read_mpdbg_ready(...)`) — single caller
   now; s5.4 will generalise it across control planes.
4. Commit on branch `mpremote_debug_command` off upstream master in the submodule
   (Q1 layout; registration into both integrations is s8.1).

## Acceptance criteria and tests

- [ ] `mpremote debug --help` lists the command with a real description (auto-derived
  from `argparse_debug`), and `mpremote help` shows it in the command list.
- [ ] `mpremote debug` with a connected device reaches the boot script and prints the
  parsed endpoint + capability dict from the `MPDBG-READY` line (no IP/port literals
  anywhere in the code path).
- [ ] house-style conformance: no new abstraction layer duplicating `exec`/`read_until`;
  single-caller helpers underscored; new args default to old behaviour.
- [ ] `state.did_action()` is called — the session does not drop into the REPL after
  the command completes.
- Adversarial: malformed `module[:method]` (empty method, extra colons) yields a
  `CommandError` with usage text, not a traceback; a target that never prints
  `MPDBG-READY` times out with a clear error naming the expected line.
