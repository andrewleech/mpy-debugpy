## Context

pre-recomposition binaries with drifted provenance (three don't even match the
submodule SHA they shipped alongside — see ROADMAP Status, STORY-3.1 note). Q2

the pinned integration SHA and publishes to GitHub Releases; the v1 board set is
**unix, rpi_pico_w (RPI_PICO_W), PYBD_SF6, ESP32_GENERIC**. Rationale and lineage:
[[20260706_recomposition.md]], decisions D5/Q2 in ROADMAP.md.

## Scope
In scope: a GitHub Actions workflow in THIS repo that (a) checks out the repo with
submodules at the pinned SHAs, (b) builds mpy-cross + the four targets with the
debug flags, (c) computes sha256 per artifact, (d) publishes artifacts to a GitHub
Release keyed on the micropython submodule SHA, and (e) regenerates/updates the
`firmware/firmware.toml` entries (repo, source_commit, flags, sha256, download_url)
replacing the four legacy entries. Local dockerised reproduction path documented.
Out of scope: capability-driven variant selection (s3.3); docs (s3.4); publishing
from the micropython fork's own CI (rejected by Q2).

## Design constraints
- Reproducibility beats convenience: an artifact must be rebuildable from
  (source_commit, recorded flags) alone. Prefer committing the flag enables to
  the board configs (guarded `#ifndef` blocks in each board's mpconfigboard.h,
  matching how unix does it in mpconfigvariant.h) over ephemeral make
  arguments. Post-phase-0 rule: such a commit must live on a NEW local-only
  fork branch registered in `mbm.toml` (mirroring `local_names_implementation`)
  and enter the integration via `mbm rebase --local` with a pinned target — a
  hand-commit on `mpy-debugpy` is force-reset away on the next rebuild. If
  make-arg injection is used instead, the exact invocation must be recorded in
  the manifest `build_flags`/notes.
- `MICROPY_PY_SYS_SETTRACE_LOCALNAMES_PERSIST` stays 0 everywhere — it corrupts
  line numbers for all compiled code ([[20260706_recomposition.md]]).
- Manifest capability fields are build intent, never behaviour claims — the
  runtime probe (`debugpy/server/debug_session.py:82 probe_capabilities`) is the
  source of truth (D4; firmware.toml header).
- Local docker builds use the host-uid/path convention:
  `docker run --rm -v "$(pwd):$(pwd)" -w "$(pwd)" --user "$(id -u):$(id -g)" <image> <cmd>`.
- Repo must stop carrying binaries in-tree (STORY-3.1 acceptance criterion still
  open) — this story deletes `firmware/<id>/` blobs once Releases exist.

## Approach sketch
Staged (revalidation 2026-07-15), so CI iteration never couples with moving pins:

1. **Compose the flag enables.** New local-only fork branch on
   `andrewleech/micropython` (e.g. `debug_board_flags`) with guarded `#ifndef`
   SETTRACE+LOCALNAMES enables in the three device boards' mpconfigboard.h
   (PERSIST untouched at 0). Push to the fork, register as a third micropython
   entry in `mbm.toml`, recompose with the pinned target (equivalence check:
   update-vs-integration diff is exactly the boards change), force-move +
   push, reset side-effect branches, bump the submodule pin here, `make test`
   green.
2. **Local dockerised builds first.** Prove all four targets build from the
   new pin locally (host-uid/path docker convention; espressif/idf image for
   esp32, ARM GCC for rp2/stm32) before any CI YAML exists — record artifact
   paths, sha256s, toolchain versions.
3. **CI workflow.** One workflow, matrix over the four targets. unix: plain
   ubuntu runner, artifact = `ports/unix/build-standard/micropython` (ELF).
   rp2: cmake, artifact = `build-RPI_PICO_W/firmware.uf2`. stm32 PYBD_SF6:
   arm-none-eabi build, artifact is the dfu (`TEXT0_ADDR = 0x08008000`,
   `boards/PYBD_SF6/mpconfigboard.mk:7`) — confirm whether PYBD deployment
   expects mboot-wrapped dfu and name the artifact accordingly. esp32:
   espressif IDF (docker image `espressif/idf:<pinned>` is the cleanest
   runner-independent route), artifact = `build-ESP32_GENERIC/firmware.bin`.
   Iterate on a short-lived branch (Q7); land on `main` once green.
4. **Release + manifest.** A final job assembles the Release (tag =
   `fw-<micropython-short-sha>`) from `{artifact, artifact.sha256}` uploads
   and emits the manifest fragment; commit the `firmware.toml` update with the
   real entries. Decide the s3.3 leftover here: drop the four `deprecated`
   legacy entries (leaning: yes — the header note plus git history suffice).
5. **Deliberate master bump (stage 2, after CI is green on current pins).**
   First default-target `mbm rebase --local` onto current `upstream/master`
   (phase-0 carried-forward item): moves every pin, re-runs the firmware test
   suites and `make test`, then re-runs the CI pipeline to a second Release —
   which is the re-runnability acceptance proof. Expect the
   `extmod/select_poll_fd.py` baseline to shift (phase-0 note).

Makefile gets a `firmware-fetch` convenience target later (s3.3 territory —
don't build selection logic here).

Learned, binding on later stages:

- mbm recreates every registered branch's commits with fresh hashes on every
  rebase (verified: identical tree/parent/author-date, committer-date +1s), so
  equivalence gates MUST be tree-diff based; commit-count/SHA gates are
  unsatisfiable.
- This rebuild hit a pdb_support/local_names conflict (py/profile.c,
  mpconfigvariant.h) that phase-0's run did not; resolved byte-identical to
  the prior integration tip and rerere-recorded, so future rebuilds replay it.
- Docker bind mounts are silently EMPTY on this dev machine's WSL distro —
  local builds use a docker cp in/out pattern (see the stm32/esp32 invocation
  records in the session log); GitHub runners are unaffected.
- unix toolchain record needs gcc/glibc versions pinned by the CI runner image
  for the sha256-reproducibility acceptance to be meaningful (review nit).
