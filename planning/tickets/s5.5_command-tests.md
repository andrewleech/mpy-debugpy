# tests: drive `mpremote debug` end-to-end (unix + QEMU PTY) through the DAP harness

Phase: EPIC-5 / STORY-5.5
Depends on: s5.3 (unix flow), s5.4 (handshake plumbing); QEMU leg additionally on mpremote PR #18327 being in the tree under test (see constraints)
Written: 2026-07-06 at HEAD 59fab8450a (micropython 18c76f7c56, micropython-lib 0d42fcd5bc; ampremote/micropython 5b459e2d57)

## Context

EPIC-1 proved the harness pattern: a fake VS Code DAP client drives a real session
against a real MicroPython process, so tests fail on behaviour regressions, not
source-string drift (repo CLAUDE.md test rule). This ticket points that same harness
at the `mpremote debug` command instead of spawning the launcher directly — proving
the command end-to-end: attach, breakpoint, step, inspect, continue, with all
endpoints handshake-derived.

## Scope

In scope: pytest coverage where the debuggee is started via `mpremote debug unix ...`
(the command under test, invoked as a subprocess); a QEMU PTY leg where feasible in
CI without hardware; assertions that no endpoint is hardcoded (the test reads the
command's re-emitted handshake and attaches to THAT).

Out of scope: hardware-in-loop device tests (STORY-6.4/s6.4); mpremote's own shell
tests in `tools/mpremote/tests/` (s5.1 covers registering one there); extension
smoke tests (s7.4).

## Design constraints

- Tests stay machine-independent: no absolute paths (repo CLAUDE.md); derive paths
  from `pytestconfig.rootpath` as `debuggee.py:91` does; `mpdebug.toml` for the test
  written to `tmp_path`, not committed at repo root.
- **QEMU PTY dependency is NOT in this repo's tree**: PR #18327 (PTY detection,
  `transport_serial.py:108-111` in ampremote's tree) exists only in
  `~/ampremote/micropython`; this repo's micropython submodule @ 18c76f7c has plain
  upstream mpremote. The QEMU leg therefore requires either registering that branch
  into the `mpy-debugpy` integration branch first (recommended, Q1/D5-consistent —
  surface as a work item at phase entry) or running against ampremote. Do not let
  the QEMU leg silently test a different mpremote than the unix leg.
- The mpremote under test must be the integration's copy
  (`micropython/tools/mpremote`), invoked hermetically (e.g. `uv run --with-editable
  micropython/tools/mpremote -m mpremote ...` or PYTHONPATH), not whatever
  `mpremote` is on PATH.
- CI budget: the unix leg runs in `make test`; the QEMU leg may be marked/skipped
  when QEMU or the PTY-capable mpremote is absent — skip with a reason string, never
  xfail-mask a real failure (lesson recorded from the EPIC-1 xfail misdiagnosis).

## Approach sketch

1. `mpremote_debuggee` fixture: write `mpdebug.toml` (unix target, firmware path
   from `MPY_DEBUG_FIRMWARE` or the standard build) into `tmp_path`; spawn
   `python -m mpremote debug unix target:main` with cwd=`tmp_path`,
   `MICROPYPATH` per `debuggee.py:108`; non-blocking stdout; parse the command's
   re-emitted `MPDBG-READY` (s5.3 contract) for host/port.
2. Attach `ThreadedServer` to that endpoint; run the EPIC-1 assertion sequence:
   setBreakpoints → configurationDone → stopped(breakpoint) → stackTrace → scopes →
   variables → continue → target completes.
3. Endpoint-hygiene test: run with `--port 0` twice concurrently; both sessions get
   distinct handshake-derived ports; assert neither test nor command output contains
   a configured endpoint.
4. QEMU leg (gated): same flow with a `kind=serial` target whose connect string is
   the QEMU PTY; requires the PTY-capable mpremote (see constraints).

## Acceptance criteria and tests

- [ ] unix flow passes in CI via `make test`: attach, breakpoint stop, step, locals
  inspect (readOnly hint per caps), continue to completion — all through
  `mpremote debug`.
- [ ] endpoints asserted handshake-derived: port requested as 0, attached port taken
  from the command's output; two concurrent sessions don't collide.
- [ ] QEMU leg passes where its prerequisites exist, and skips (with reason) where
  they don't; the skip reason names the missing prerequisite.
- Adversarial: kill the debuggee mid-session — the command exits non-zero with a
  clear message and the test asserts no orphan processes; corrupt `mpdebug.toml` →
  resolver error surfaces through the command's exit code/stderr.
