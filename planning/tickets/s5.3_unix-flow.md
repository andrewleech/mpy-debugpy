# mpremote: `debug` unix flow end-to-end (ensure firmware → launch → parse handshake → attach)

Phase: EPIC-5 / STORY-5.3
Depends on: s5.1 (skeleton), s5.2 (target resolution), STORY-3.1 (manifest, done), EPIC-1 (done)
Written: 2026-07-06 at HEAD 59fab8450a (micropython 18c76f7c56, micropython-lib 0d42fcd5bc; ampremote/micropython 5b459e2d57)

## Context

The unix port is the lowest-friction proof of the whole resolve→ensure→launch→attach
model: no device, no sync, subprocess stdout as the control plane. Everything this
flow needs already exists and is test-proven in the wrapper repo — this ticket moves
the orchestration from pytest fixtures into the command. Bound by D1 (command owns
orchestration) and the handshake-only rule: the endpoint comes from the `MPDBG-READY`
line, never from config.

## Scope

In scope: `kind=unix` branch of `do_debug`: locate/ensure a debug-capable unix
binary; set `MICROPYPATH`; spawn the boot script as a subprocess; read stdout until
the `MPDBG-READY` line; report endpoint + caps (and, for now, leave the subprocess
running for an external DAP client to attach — the command IS the launcher).
`--port` passthrough (default 0 = auto).

Out of scope: acting as a DAP proxy or client itself; serial/network control planes
(s5.4/EPIC-6); firmware CI publishing (s3.2); VS Code integration (s7.1, which
spawns this command and reads the same line).

## Design constraints

- No literal ports/IPs: `--port` default 0; the reported endpoint is authoritative
  (public_api.py port-0 fallback means the handshake may differ from the request).
- The launcher's stdout is line-oriented but arrives in non-blocking chunks — parse
  by accumulating and splitting, exactly like `_read_mpdbg_ready`; do not readline()
  on a blocking pipe (the banner + prompt output precedes the handshake).
- Respect capability truth: after parsing caps, validate the target's `requires`
  list (s5.2) — mismatch is a hard error before reporting success.
- The subprocess must be cleaned up on error paths (terminate→kill ladder as in
  `debuggee.py:219-233`).

## Approach sketch

`_do_debug_unix(state, target, module, method, port)`:
1. Resolve binary: env override → target firmware → error with `make firmware-unix`
   hint.
2. Build env (`MICROPYPATH` covering project src + debugpy module path — for the
   command running outside this wrapper repo, the debugpy module location becomes a
   config/packaging question: see Open questions).
3. Spawn `[binary, launcher, module, method, str(port)]`, non-blocking stdout.
4. `_read_mpdbg_ready(proc, timeout)` → `{host, port, caps}`; validate caps against
   `requires`. NOTE: on unix `_detect_host()` (launcher `:34-54`) always reports
   `host: "0.0.0.0"` — a bind address, not a connectable one. Translate to
   `127.0.0.1` before emitting (rule owned by s5.4: the tooling never emits a raw
   `0.0.0.0` endpoint). Then print a stable one-line summary (what s7.1's extension
   parses — machine-friendly: re-emit the `MPDBG-READY` JSON with the resolved host).
5. Leave the process attached to the console (stream its output) until it exits or
   Ctrl-C; on Ctrl-C terminate the subprocess.

## Acceptance criteria and tests

- [ ] `mpremote debug unix target:main` (with `mpdebug.toml` present) reaches a
  breakpoint set by a DAP client with zero hand configuration.
- [ ] port is auto (0) end-to-end; the reported port comes from the handshake; grep
  the implementation for port literals — none besides the argparse default of 0.
- [ ] missing binary produces the build hint, not a traceback; handshake timeout
  names the expected line and dumps captured stderr.
- [ ] subprocess is reaped on every failure path (test with a binary that exits
  immediately and one that never prints the handshake).
- Adversarial: launcher printing a corrupt/duplicate `MPDBG-READY` line (must error,
  matching the "exactly one" contract tested at `test_epic1_foundations.py:51`);
  caps missing a required capability → hard error naming the capability.
