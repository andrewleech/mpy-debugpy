# mpremote: fold the DAP monitor into `do_debug` as `--dap-log`

Phase: EPIC-6 / STORY-6.3
Depends on: s5.1 (do_debug skeleton)
Written: 2026-07-06 at HEAD 59fab8450a (micropython 18c76f7c56, micropython-lib 0d42fcd5bc)

## Context

Today DAP traffic inspection is a separate hand-run proxy the user must insert
between VS Code and the device (extra port, extra task, one of the old wrapper's
nine launch configs pointed at it). Since `do_debug` now owns the attach path for
every transport, logging belongs there as a flag, not a standalone tool.

## Scope

In scope: a `--dap-log[=FILE]` flag on `mpremote debug` that records every DAP
message (direction, timestamp, decoded JSON) for ANY transport, and retirement of
the standalone monitor as the documented workflow.
Out of scope: protocol analysis/statistics; changing what the monitor logs beyond
direction+timestamp+payload.

## Files and anchors

Verified at the stamped SHAs:

- **Existing implementation** — `micropython-lib/python-ecosys/debugpy/dap_monitor.py`
  (199 lines, top level of the debugpy package dir, NOT under `debugpy/`). Host-side
  CPython: a threaded TCP proxy (`DAPMonitor`, listens 5679, forwards to 5678,
  hardcoded localhost defaults at :13). Uses `threading` + blocking sockets — fine on
  host, irrelevant to device. Its useful part is the message decode/log formatting,
  not the socket plumbing.
- **Integration point** — where `do_debug` (s5.1) holds the client↔server byte
  stream. Two shapes exist per transport: direct TCP attach (network/unix) and the
  s6.1 localhost↔serial bridge (which is *already* a proxy — `--dap-log` there is
  just a tee in the existing pump).
- The DAP wire format is Content-Length-framed JSON
  (`micropython-lib/python-ecosys/debugpy/debugpy/common/messaging.py`) — the logger
  must parse frames, not lines.

## Design constraints

- Works uniformly across transports; the flag must not change timing enough to mask
  the races it exists to debug (log asynchronously or after-the-fact, don't block
  the pump on disk I/O).
- When VS Code attaches directly to the device TCP endpoint (not through any
  mpremote-owned pump), logging requires interposing a local proxy — that is exactly
  what dap_monitor already is; reuse its proxy shape for that one case, now spawned
  and wired automatically by the flag instead of by hand.
- House style: this lands in the mpremote `debug` command branch (Q1).

## Approach sketch

1. Extract a small `DapLogger` (frame parser + `{ts, dir, msg}` JSONL writer) usable
   as a tee.
2. Bridge/pump transports (s6.1 serial bridge; unix subprocess if piped): tee
   in-line.
3. Direct-TCP attach: `--dap-log` inserts a localhost proxy (dap_monitor's shape,
   port auto-assigned, endpoint substituted into the handshake result handed to the
   client/VS Code) so the flag works without the user re-pointing anything.
4. Remove the standalone monitor from documented workflows; keep the module until
   EPIC-8 docs land, then deprecate.

## Acceptance criteria and tests

- [ ] `mpremote debug <target> --dap-log` writes a JSONL log for unix AND network
  transports (serial via the s6.1 bridge when that lands).
- [ ] Log contains both directions with correct framing (a multi-frame TCP segment
  and a frame split across segments are both logged as complete messages —
  adversarial framing test with a scripted byte stream).
- [ ] With the flag, the EPIC-1 behavioural test suite still passes (logging does
  not perturb the session).
- [ ] No hardcoded ports: the interposed proxy port is auto-assigned and flows
  through the normal handshake substitution.
