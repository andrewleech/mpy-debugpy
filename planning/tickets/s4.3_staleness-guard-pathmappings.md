# mpremote+debugpy: Attach-time staleness guard + generated absolute pathMappings

Phase: EPIC-4 / STORY-4.3
Depends on: s4.2 (sync record + manifest, for no-mount mode) or s4.1 alone (mount mode); consumed by s5.3/s5.4 and s7.1
Written: 2026-07-06 at HEAD 59fab8450a (micropython 18c76f7c56, micropython-lib 0d42fcd5bc; ampremote/micropython 5b459e2d57)

## Context

Two chronic failure modes from the old wrapper: debugging stale device code without
knowing (silent `cp -r` skipped, or interrupted), and hand-written `pathMappings` in
nine `launch.json` configs with relative/ambiguous roots that break in multi-root
workspaces. This ticket makes staleness loud and pathMappings generated — both derived
from the sync/mount record the command already has, never typed by a user.

Constraining decisions: D2 (mount mode makes staleness structurally impossible — the
guard is trivial there), D1 (all logic in the command, not the editor).

## Scope

In scope: (a) a pre-attach check comparing device file hashes against local for the
files the session will debug, hard-warning (or blocking, per config) on mismatch;
(b) generation of absolute `pathMappings` from the sync/mount record, passed to the DAP
attach request.
Out of scope: performing the sync itself (s4.2); the editor-side plumbing that forwards
the generated mappings to the debugpy attach session (s7.1 consumes them).

## Design constraints

- Mount mode: source is live by construction; guard reduces to asserting the mount is
  active, and pathMappings map the mount root (`/remote`) to the absolute local root.
- No-mount mode: guard checks the manifest hashes of (at minimum) the entry module and
  its project-local imports against local files; mismatch → hard warning naming the
  stale files before attach, blocking if `strict` is configured.
- Generated pathMappings are ABSOLUTE local paths (multi-root safe) — this is an
  explicit acceptance criterion from the roadmap; never emit `${workspaceFolder}` or
  relative roots.
- The user never writes pathMappings by hand anywhere in the v1 flow.

## Approach sketch

1. In the command's attach step (s5.3/s5.4), after ensure+sync: build pathMappings from
   the sync record (no-mount) or mount root map (mount).
2. No-mount: fetch device-side hashes for the session's file set (single exec snippet
   returning `{path: sha256}`), compare to fresh local hashes — NOT to the manifest
   alone, so a hand-edited or interrupted device state is still caught.
3. On mismatch: print the stale file list + the exact resync command; `strict` config
   blocks the attach.
4. Emit the final attach `pathMappings` JSON; s5.3 injects it into the DAP attach
   request; s7.1 forwards it into the VS Code session config.

## Acceptance criteria and tests

- Deliberately staled device file (edit local after sync) → warning names that file
  before the session starts; with `strict`, attach is refused.
- Mount-mode session on a multi-root-style layout → breakpoints bind with the generated
  absolute mappings (no `${workspaceFolder}`, no relative paths in the wire request).
- Round-trip property: for every synced file,
  `_filename_as_debugger(_filename_as_debugee(local))` is identity under the generated
  mappings (drives out the prefix-swap edge cases; run against `pdb_adapter.py:142/155`).
- Adversarial: device file matches the manifest but differs from local (manifest lies)
  → still detected, because the guard hashes the device files, not the manifest.
