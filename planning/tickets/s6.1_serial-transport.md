# mpremote: serial device transport — localhost↔serial DAP bridge for multi-CDC boards

Phase: EPIC-6 / STORY-6.1
Depends on: s5.1 (do_debug skeleton), s5.4 (handshake plumbing), SPIKE-2.2 verdict (D3), Q3 DECIDED
Written: 2026-07-06 at HEAD 59fab8450a (micropython 18c76f7c56, micropython-lib 0d42fcd5bc; ampremote/micropython 5b459e2d57)

## Context

SPIKE-2.2 (`../SPIKE-2.2-serial-dap.md`) found DAP-over-serial is board-dependent:
clean on boards exposing a second USB CDC interface (native-USB: RP2040, ESP32-S2/S3,
some STM32/nRF52), high-risk on single-UART boards (CP2102/CH340 bridge boards —
most classic ESP32 devkits). Decisions binding this ticket:

- **D3 (2026-07-05):** serial DAP is first-class only on multi-CDC boards; network is
  the primary device transport.
- **Q3 DECIDED (2026-07-06):** v1 ships network-only for single-UART boards. The
  single-UART framing/escaping work is *retained but deferred* — see the appendix,
  which preserves the spike's design detail so a later follow-up spike starts warm.

So this ticket builds the serial path **only for the multi-CDC subset**, gated on a
runtime probe (never inferred from a board name — same philosophy as STORY-1.2 /
`caps` in the MPDBG-READY handshake).

## Scope

In scope:
- A device-side stream adapter (`StreamTransport`, sketch below) letting the debugpy
  server run its DAP channel over a second CDC interface instead of a TCP socket.
- A host-side localhost↔serial DAP bridge owned by `mpremote debug`: VS Code attaches
  to a localhost TCP port; the bridge shuttles bytes to/from the dedicated CDC
  interface.
- A runtime probe for "does this board expose a second CDC interface routed to DAP",
  surfaced in the handshake `caps` (e.g. `serial_dap: true/false`).

Out of scope (deferred, with detail retained in the appendix):
- Single-UART framing/escaping (mux DAP with REPL/stdout on one stream) — future
  follow-up spike, per Q3.
- Any change to `messaging.py` framing (Content-Length + JSON stays as-is; the spike
  confirmed only the byte-stream source needs to change).

## Files and anchors

Verified at the stamped SHAs:

- **What the server assumes of "the socket"** (the adapter must match all of it):
  `micropython-lib/python-ecosys/debugpy/debugpy/common/messaging.py` —
  `self.sock = sock` (:11), `send(header+content)` (:51), `recv(...)`
  (:93/:114/:144, must raise `OSError` errno 11/35 for no-data and return `b""` on
  close), `close()` (:179). `settimeout` is mutated at runtime from OUTSIDE the
  channel: `public_api.py:62` (0.001 after initialize) and multiple sites in
  `debugpy/server/debug_session.py` (0.001/0.5/1.0/None at protocol points). The
  adapter must be socket-shaped including the settimeout state machine.
- **Trace-callback frequency constraint:** `debug_session.py` trace function calls
  `process_pending_messages()` on every call/line/return event; paused busy-poll is
  ~10 ms (`pdb_adapter.py` wait_for_continue). `recv()` must be cheap
  (`select.poll(0)` + readinto).
- **Non-blocking-poll precedent on device:** mount's `fs_hook_code` in this repo's
  mpremote — `micropython/tools/mpremote/mpremote/transport_serial.py:440` (code
  string), `mount_local` (:319) — proves `select.poll` on a stream works mid-execution.
- **Reconnect/rfc2217 primitives are NOT in this repo's mpremote** (plain upstream at
  18c76f7c56): no reconnect or rfc2217 code exists under
  `micropython/tools/mpremote/`. They live only in ampremote (5b459e2d57):
  `mpremote/commands.py:80` (`do_reconnect`), `mpremote/main.py:548-559` (reconnect
  config/env default), `mpremote/repl.py:6` (`_is_rfc2217`). Per Q1, `do_debug`
  develops on a branch consumed by BOTH trees — so this ticket must treat reconnect
  as an *optional* integration (feature-detect, don't hard-depend), or the branch
  won't apply to upstream master. Board-reset handling can use ampremote's reconnect
  when present and degrade to a clear error upstream.

## Design constraints

- No background thread on the device (server is trace-callback-driven) — the adapter
  is polled, never threaded, on-device. The HOST bridge may thread freely (CPython).
- Capability truth: `serial_dap` reported by runtime probe in `caps`; a board without
  a second CDC never advertises it. Mismatch between claim and probe is a hard error
  (STORY-3.3 rule).
- Serial device references use `/dev/serial/by-id/<name>`, never raw `/dev/ttyACMx`.
- Upstream-mergeable: device-side adapter lands in micropython-lib debugpy; bridge
  lands in the mpremote `debug` command branch (Q1: one branch, two consumers).

## Approach sketch

1. Device side: implement `StreamTransport` per the spike's sketch —
   `recv(n)`/`send(data)`/`settimeout(s)`/`close()` over reader/writer stream objects
   with `select.poll`; `debugpy.listen()` grows a way to accept a pre-existing stream
   pair instead of binding TCP (keep the TCP path untouched).
2. Probe: at boot-script time, detect a usable second CDC interface (port-specific;
   start with RP2040 `machine.USBDevice` / rp2 TinyUSB CDC #2) and report
   `serial_dap` in `caps`.
3. Host side: `mpremote debug` opens the second CDC device node (by-id, second
   interface suffix), binds a localhost listener, and pumps bytes both ways;
   endpoint goes into the same MPDBG-READY-derived attach flow as TCP (s5.4's
   control/data-plane split — control plane stays on the primary REPL channel).
4. First target: **rpi_pico_w** (native USB, in the Q2 board set). PYBD_SF6 (STM32
   multi-CDC, `MICROPY_HW_USB_CDC_NUM`) second. ESP32_GENERIC is single-UART —
   explicitly network-only in v1.

## Acceptance criteria and tests

- [ ] On rpi_pico_w: `mpremote debug <serial-target>` reaches a breakpoint with only
  the connect string (no IP anywhere), DAP flowing over the second CDC via the bridge.
- [ ] On a single-UART board (ESP32_GENERIC): the probe reports `serial_dap: false`
  and the command falls through to network transport with a clear message (no hang,
  no misleading error).
- [ ] Adversarial: kill the bridge mid-session — device recovers to a usable state
  (trace uninstalled or session cleanly closed), no wedged CDC.
- [ ] Board reset mid-session: with ampremote's reconnect present, session ends with
  a reconnect-and-report; with plain upstream, a clear error (feature-detected).
- [ ] Host-side unit tests for the bridge byte-pump (no hardware: loopback PTY).

## Appendix — retained single-UART framing detail (deferred by Q3, do not lose)

From SPIKE-2.2 §2, preserved for the future follow-up spike:

- One shared byte stream ⇒ DAP frames need collision-proof framing against arbitrary
  program output. Mount's `0x18` sentinel scheme is device-initiated, half-duplex,
  Ctrl-C-disabled and desync-tolerant only because fs-RPC is rare/short; DAP would
  run a scanner permanently, host-initiated, full-session.
- Escaping must cover ALL device output paths; C-level tracebacks/panic dumps bypass
  any Python `sys.stdout` wrapper and can corrupt a frame; `recv_message` trusts
  Content-Length and does not resynchronise on garbage — a resync strategy
  (SLIP/COBS-style framing with a magic delimiter) is mandatory, not optional.
- Poll cost at trace-callback frequency is proven feasible by mount's
  `RemoteCommand` poll shape; the unproven part is coexistence with unconstrained
  `print()` at 115200–921600 baud without byte loss.
- Unix-pipe prototyping validates only the socket→stream adapter (§1), not the
  multiplexing problem — do not present it as evidence for single-UART viability.
