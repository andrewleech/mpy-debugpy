## Context
The legacy variant set encodes capabilities in names, and at least one name lies:
`unix_settrace_set_local` claims local write-back that no branch of any lineage
implements (`frame._set_local` exists nowhere — re-confirmed at `18c76f7c`,
[[20260706_recomposition.md]]; D4). BACKGROUND.md documents this as the founding
failure mode. Tooling must select firmware by the capability a session needs and
treat a claimed-vs-probed mismatch as a hard error.

## Scope
In scope: reduce the manifest to the real variant set (per board: one debug build
with settrace + LOCALNAMES; the old save_names/local_xx/set_local split disappears
— LOCALNAMES is simply on in all debug builds); add capability-driven selection to
`launcher/firmware.py` (`select --need settrace,save_names` style, resolving
against manifest intent) and a post-attach cross-check that compares the manifest's
claimed capabilities to the probed `caps` from the MPDBG-READY handshake, failing
hard on mismatch.
Out of scope: building/publishing artifacts (s3.2); surfacing capability in the VS
Code UI (s7.2); the probe itself (DONE, STORY-1.2).

## Design constraints
- Capability is probed at runtime, never inferred from names (D4; firmware.toml
  header; CLAUDE.md "Firmware" section). Manifest capabilities are *intent*;
  the probe is *truth*; a mismatch is a bug in the manifest or the build and must
  abort the session with both values printed.
- `save_names` in the caps dict maps to `MICROPY_PY_SYS_SETTRACE_LOCALNAMES`
  (not `_SAVE_NAMES`) on the current branch — keep the probe-key spelling stable
  even though the macro name changed, since EPIC-1's handshake tests assert it.
- No variant may promise `set_local` until firmware implements it; when that day
  comes the probe, not a new variant name, unlocks the UX (D4).

## Approach sketch
1. Rewrite the manifest to the collapsed per-board set (lands with s3.2's first
   Release; until then this ticket can stage the schema change with the legacy
   entries marked `deprecated = true` rather than deleted, preserving provenance).
2. `firmware.py select`: filter variants by port/board + required caps ⊆ claimed
   caps; exactly-one match returns its artifact path (fetching if absent);
   zero/many matches is a clear error listing candidates.
3. Mismatch guard: a small helper (importable by the future `mpremote debug`
   flow, STORY-5.3) that takes (manifest claimed caps, probed caps) and raises on
   any claimed-true-but-probed-false key. Wire it into the unix launch path used
   by the tests as the first consumer.

## Execution outcome (2026-07-15)

Implemented as `launcher/firmware.py select` (`--need cap[,cap]`,
`--port`/`--board` filters, exit 2 for usage/config errors, 1 for
no-unique-match) resolving strictly `is True` claims against
`KNOWN_CAPABILITIES = (settrace, save_names, set_local, f_back)`, and
`launcher/capcheck.py` `check_capabilities(claimed, probed)` raising
`CapabilityMismatch` naming every claimed-true-but-probed-false key. Wiring:
opt-in `claimed_capabilities`/`probed_capabilities` fixtures in
`tests/fixtures/debuggee.py` (env var `MPY_DEBUG_CLAIMED_CAPS` or fixture
param); the MPDBG-READY parser moved from `test_epic1_foundations.py` into the
shared fixture module (the ticket's "fixture already parses it" premise was a
minor factual gap — it lived in the test file).

Notes carried to s3.2/s5.3: `select` returns a pre-existing local artifact
without re-verifying its sha256 (fetch does verify — decide at s3.2 whether
select should too); `KNOWN_CAPABILITIES` must stay in lockstep with
`debugpy.get_capabilities()` by hand; `verify` with no args now fails against
the all-deprecated binary-less manifest (expected pre-s3.2 state); network
errors during fetch surface as tracebacks rather than clean errors; the guard
is exercised by its own tests, not yet by the broader DAP session flow (that
wiring is s5.3's resolve-ensure-attach path).
