# mpremote+wrapper: Ensure debugpy package present on device, cross-compiled and hash-cached

Phase: EPIC-4 / STORY-4.1
Depends on: STORY-2.1 (DONE — D2 conditional yes), mpremote transport with `verify_hash` (ampremote lineage; see Open questions)
Written: 2026-07-06 at HEAD 59fab8450a (micropython 18c76f7c56, micropython-lib 0d42fcd5bc; ampremote/micropython 5b459e2d57)

## Context

Mount is the mainline source-delivery mechanism (D2, `../SPIKE-2.1-mount.md`), so the
only thing that must physically live on the device filesystem is the debugpy package
itself (a mounted debugpy would add fs-RPC latency to every trace callback and dies with
the serial link). Today the old wrapper re-ran `compile_debugpy.py` + `mip install` +
`cp -r` on every run with no staleness detection. This ticket makes package install a
cached, verified, one-shot step: cross-compile to `.mpy`, install once, skip when
nothing changed.

Constraining decisions: D2 (mount mainline → this ticket is the ONLY unconditional part
of EPIC-4), D5 (canonical lineage), Q1 (mpremote work lives on one `andrewleech/micropython`
branch consumed by both ampremote and this repo).

## Scope

In scope: a reusable "ensure debugpy on target" step — cross-compile the debugpy package
sources to `.mpy`, compute a cache key, compare against an on-device marker, install only
on mismatch, verify each written file by device-side sha256.
Out of scope: syncing USER source (s4.2 fallback / mount mainline); attach-time staleness
of user source (s4.3); wiring into `do_debug` (s5.3 consumes this as a function).

## Files and anchors

- Package to install: `micropython-lib/python-ecosys/debugpy/debugpy/` — exactly 8
  modules: `__init__.py`, `public_api.py`, `common/{__init__,constants,messaging}.py`,
  `server/{__init__,debug_session,pdb_adapter}.py`. `manifest.py` at the package root is
  `package("debugpy")` — the module list can be derived by globbing the package dir, no
  hand-kept JSON manifest (the old repo's `debugpy.json`/`debugpy_mpy.json` pattern is
  what this replaces).
- mpy-cross build: `Makefile` target `mpy-cross` (`make -C micropython/mpy-cross`);
  binary lands at `micropython/mpy-cross/build/mpy-cross`. Version string for the cache
  key via `mpy-cross --version`.
- Hash-verified write: `fs_writefile(..., verify_hash=True)` exists ONLY in the
  ampremote lineage — `~/ampremote/micropython/tools/mpremote/mpremote/transport.py:292`
  (writefile) and `:194` (readfile). This repo's micropython submodule
  (`micropython/tools/mpremote/mpremote/transport.py:154`) has the plain
  `fs_writefile(dest, data, chunk_size=256, progress_callback=None)` — NO verify_hash.
  See Open questions.
- Existing hash primitive on device: upstream mpremote computes device-side sha256 via
  `hashlib` in the helper snippets ampremote's #18436 branch adds; reuse, don't reinvent.

## Design constraints

- Never trust names: the installed-package marker records content hashes, not versions.
- No behaviour change to the debugpy package itself here — install-side only.
- Cache key = (sha256 over sorted source files, mpy-cross version string, mpy-cross
  flags e.g. `-O2`, march if set). Any component changes → full reinstall.
- Serial-transport friendly: must work over raw REPL with nothing but `state.transport`
  primitives, since `do_debug` (s5.1) will call it before the session starts.
- Follow mpremote house style (extend existing primitives; new args default to old
  behaviour).

## Approach sketch

1. Host side: glob the package dir, cross-compile each `.py` → `.mpy` into a host cache
   dir keyed by the cache key (so switching firmware/flags doesn't thrash).
2. Compute the install manifest: `{device_path: sha256(mpy_bytes)}` + the cache key.
3. On device, read `:/lib/.debugpy-install.json` (or absent). If key matches, done —
   zero file transfers.
4. Else write each `.mpy` under `:/lib/debugpy/...` with `verify_hash=True`, then write
   the marker file last (so an interrupted install re-runs).
5. Expose as a function importable by the future `do_debug` (Q1 branch) and callable
   standalone for tests.

## Acceptance criteria and tests

- First run against a wiped device installs all modules; `import debugpy` succeeds on
  target.
- Second run with unchanged sources performs no `fs_writefile` calls (assert via
  transport call spy / DAP-less unit test).
- Touching one source file reinstalls (at minimum) that module and updates the marker.
- Changing mpy-cross version or flags forces full reinstall.
- A corrupted transfer (simulated hash mismatch) fails loudly, and the marker is NOT
  updated.
- Adversarial: kill the install mid-way; next run must detect the stale/absent marker
  and reinstall rather than assume success.

## Open questions

- `verify_hash` is not on this repo's mpremote (only ampremote's #18436 branch). Per Q1
  the debug tooling develops on one `andrewleech/micropython` branch — decide whether to
  (a) register the #18436 branch into this repo's `mpy-debugpy` integration too, or
  (b) make verify_hash optional (fallback: write then `fs_hashfile`-style re-read
  compare). (a) is consistent with D5/Q1 and recommended.
- Install path `:/lib/debugpy` vs frozen-into-firmware for the CI builds (s3.2) — if
  s3.2 freezes debugpy into device firmware, this ticket's runtime probe should detect
  the frozen copy and skip install entirely.
