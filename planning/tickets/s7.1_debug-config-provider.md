# extension: Register `micropython` debug type whose resolver shells out to `mpremote debug` and starts a debugpy attach

Phase: EPIC-7 / STORY-7.1
Depends on: EPIC-5 usable (`mpremote debug` launches and prints the handshake; tickets s5.1–s5.4), STORY-4.3 (generated absolute pathMappings)
Written: 2026-07-06 at HEAD 59fab8450a (micropython 18c76f7c56, micropython-lib 0d42fcd5bc)

## Context

The end state is F5 one-keypress debugging of MicroPython from VS Code (Q4 DECIDED
2026-07-06: the extension IS in v1 scope — it is the primary marketing win). All
orchestration logic lives in the `mpremote debug` command (settled decision D1,
`ROADMAP.md` "Design position"); the extension exists only to deliver the keypress.
Today the attach flow is manual: `.vscode/launch.json` prompts the user to type the
host/port they read off the `MPDBG-READY` line — exactly the friction this ticket
removes. See `ROADMAP.md` EPIC-7 rationale and the design position summary.

STORY-7.3 (collapse launch.json/tasks.json, delete `MP_DEBUG_VARIANT` and
problemMatcher hacks — hacks that live in the OLD Josverl wrapper repo, largely
already absent here) is intentionally inline-only in the roadmap (trivial) and lands
alongside this ticket.

## Scope

In scope: a VS Code extension that registers debug type `micropython` with a
`DebugConfigurationProvider`; on F5 it spawns `mpremote debug <target> ...`, parses
the single `MPDBG-READY {json}` handshake line from its stdout, and starts an
`ms-python.debugpy` attach session with the reported host/port and generated
**absolute** pathMappings. Extension packaging (`package.json` contributes +
`extensionDependencies`), activation events, and clear failure surfacing.

Out of scope: target picking UI (s7.2), extension-host tests (s7.4), any firmware
selection / source sync / endpoint discovery logic (lives in the command, D1), and a
TypeScript DAP adapter (never — see constraints).

## Files and anchors

- `launcher/mpy_launch_debugpy.py:90` — the handshake the resolver parses:
  `print("MPDBG-READY " + json.dumps({"host": actual_host, "port": actual_port, "caps": caps}))`.
  Exactly one line, valid JSON, prefix `MPDBG-READY `. Payload fields: `host` (str),
  `port` (int), `caps` (dict). Note `host` may be `"0.0.0.0"` (unix, or device with
  no reportable address — `_detect_host()`, `:34-54`); the resolver must translate
  `0.0.0.0` to a connectable address (localhost for unix targets; the command's
  control plane supplies the real device address otherwise — s5.4).
- `caps` dict fields (verified against
  `micropython-lib/python-ecosys/debugpy/debugpy/server/debug_session.py:82`
  `probe_capabilities()`): `settrace`, `save_names`, `set_local`, `f_back` — all
  booleans, probed at runtime, never inferred from names.
- `.vscode/launch.json:14-24` — the manual attach config this replaces: type
  `debugpy`, request `attach`, `connect: {host, port}`,
  `pathMappings: [{localRoot: "${workspaceFolder}/src", remoteRoot: "."}]`,
  `justMyCode: false`. The generated config keeps this shape but with concrete
  values and ABSOLUTE `localRoot`/`remoteRoot` (relative `remoteRoot: "."` is the
  multi-root ambiguity STORY-4.3 fixes; consume its generated map).
- New code: `extension/` in this repo (see Open questions) — `package.json`,
  `src/extension.ts` (activate + provider), `src/handshake.ts` (parser, unit-testable).

## Design constraints

- D1: the extension must NOT own firmware selection, source sync, target launch, or
  endpoint discovery. It shells out to `mpremote debug` and attaches. Anything the
  extension "needs to know" it gets from the command's stdout handshake.
- A `resolveDebugConfiguration` implementation CANNOT change a config's `type`:
  return `undefined` after calling `vscode.debug.startDebugging(folder, attachConfig)`
  where `attachConfig.type === "debugpy"` — the return-undefined-and-startDebugging
  pattern. Do not try to morph the `micropython` config into a `debugpy` one in place.
- Declare `ms-python.debugpy` in `extensionDependencies`; if unavailable at resolve
  time, fail with an actionable message (install link), not a stack trace.
- Do NOT write a TS DAP adapter: the on-target debugpy server already speaks DAP
  over TCP; VS Code's Python debugger attaches to it directly.
- Endpoints always come from the handshake, never from user config (roadmap design
  position; STORY-5.4 AC).
- pathMappings must be absolute and multi-root safe (STORY-7.1 AC; STORY-4.3).
- Honest capability surfacing: never promise local editing (`set_local` is false on
  all current firmware — D4); locals arrive readOnly from the server (STORY-1.3).

## Approach sketch

`activate()` registers a `DebugConfigurationProvider` for type `micropython` and a
`vscode.debug.registerDebugConfigurationProvider(..., Dynamic)` variant for the F5
no-launch.json case. `resolveDebugConfiguration(folder, config)`:

1. Determine target (config attribute, else the picker state from s7.2, else default).
2. Spawn `mpremote debug <target> <module[:method]> --handshake-only`-style
   invocation as a child process (exact CLI per s5.1; keep the child alive — it owns
   the session), buffering stdout line-by-line.
3. On the `MPDBG-READY ` line: `JSON.parse` the remainder; map `0.0.0.0` →
   `127.0.0.1` for unix; build the attach config (type `debugpy`, request `attach`,
   `connect`, generated absolute `pathMappings`, `justMyCode: false`).
4. `vscode.debug.startDebugging(folder, attachConfig)`; return `undefined`.
5. Wire child-process lifetime to the debug session (`onDidTerminateDebugSession` →
   terminate child; child exit before handshake → surface stderr in an error toast).
6. Timeout on the handshake (mirror `WAIT_FOR_CLIENT_TIMEOUT_S = 30` on the server
   side) with a clear error including captured output.

Keep the handshake parser a pure function for s7.4's tests.

## Acceptance criteria and tests

- [ ] F5 on a unix target launches an attach session that stops at a breakpoint with
  zero hand configuration (no typed host/port anywhere).
- [ ] Generated pathMappings are absolute; correct in a multi-root workspace.
- [ ] With ms-python.debugpy missing, resolve fails with a clear actionable message.
- [ ] Killing/ending the debug session terminates the spawned `mpremote debug` child.
- [ ] Adversarial: garbage before/after the handshake line on stdout does not break
  parsing (only the `MPDBG-READY `-prefixed line is consumed); a child that exits
  without emitting a handshake produces a diagnostic error, not a hang.
- Environment: needs a built unix firmware (`make firmware-unix`) and the installed
  `ampr`/`mpremote debug` command (EPIC-5).
