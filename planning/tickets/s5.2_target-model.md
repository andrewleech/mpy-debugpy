# mpremote: named-target model (`mpdebug.toml`) and target resolution

Phase: EPIC-5 / STORY-5.2
Depends on: s5.1 (command skeleton), STORY-3.3 (real variant set ŌĆö capability names)
Written: 2026-07-06 at HEAD 59fab8450a (micropython 18c76f7c56, micropython-lib 0d42fcd5bc; ampremote/micropython 5b459e2d57)

## Context

Today's friction is config scatter: nine `launch.json` attach configs with hardcoded
IPs, an `MP_DEBUG_VARIANT` env var, and per-transport launcher invocations (ROADMAP
"The current friction"). The named-target model replaces all of it: a project-level
`mpdebug.toml` maps a target name to `kind = unix | serial | network`, a connect
string, and a capability requirement; `mpremote debug <name>` resolves the rest.
Bound by D1 (CLI owns orchestration), D4 (capability from the runtime probe, never
from names), and the global rule that serial devices are referenced by stable
`/dev/serial/by-id/<name>` paths because raw `/dev/ttyACMx` nodes renumber.

## Scope

In scope: the `mpdebug.toml` schema; a resolver that turns a target name into
(transport kind, connect string / binary path, firmware requirement, target
module:method defaults); discovery of the config file (cwd upward, like git); clear
errors for missing file / unknown name / ambiguous target; `mpremote debug` with no
target listing the available names.

Out of scope: acting on the resolution (s5.3 unix, EPIC-6 device flows); firmware
fetch/build itself (`launcher/firmware.py`, STORY-3.x); VS Code target picker
(STORY-7.2 reads the same file ŌĆö schema changes after 7.2 starts need coordination).

## Design constraints

- Serial connect strings in examples and docs use `/dev/serial/by-id/...` (stable
  across replug); accept any pyserial-valid device but warn on `/dev/tty*` paths.
- A target's `requires` (capability list) is checked against the RUNTIME probe from
  the `MPDBG-READY` caps dict after launch ŌĆö a mismatch between required and probed
  capability is a hard error (STORY-3.3 acceptance), not a warning.
- No IP addresses or ports in `mpdebug.toml` for network targets: the device reports
  its own endpoint via the handshake (design position). A network target names the
  serial control-plane device (for the pre-IP handshake) or relies on default
  connect; it never names the debug endpoint.
- Errors are `CommandError` with actionable text (mpremote's existing error type in
  `commands.py`), never tracebacks.

## Approach sketch

Schema (draft to refine during implementation):

```toml
[target.unix]
kind = "unix"
firmware = "unix-settrace-localnames"   # firmware.toml variant id or "system"
program = "target:main"                  # default module:method

[target.pico]
kind = "serial"
device = "/dev/serial/by-id/usb-MicroPython_Board_in_FS_mode_XXXX-if00"
requires = ["settrace", "save_names"]

[target.esp32]
kind = "network"
device = "/dev/serial/by-id/usb-Espressif_USB_JTAG..."  # control plane only
requires = ["settrace"]
```

Resolver: `_resolve_target(name) -> Target` dataclass-like record; search
`mpdebug.toml` from cwd upward; no name given + exactly one target ŌåÆ use it; no name
+ several ŌåÆ `CommandError` listing names ("ambiguous target; choose one of: ...");
unknown name ŌåÆ error listing names. Keep parsing tolerant of unknown keys (forward
compatibility with s7.2's picker metadata).

## Acceptance criteria and tests

- [ ] a named target resolves to transport kind + connect string + firmware
  requirement; `program` default flows through to the launcher argv.
- [ ] serial connect strings using `/dev/serial/by-id/...` pass through untouched;
  a `/dev/ttyUSB0`-style path resolves but warns.
- [ ] absent config, unknown name, and ambiguous no-name cases each produce a
  one-line actionable `CommandError` (test asserts message content, not type only).
- [ ] `requires` capabilities are validated against the probe keys list; an unknown
  capability name in the file is an error at resolve time (catches typos before a
  device is touched).
- Adversarial: duplicate target names in the file; `kind` outside the enum; an
  `mpdebug.toml` in a parent directory being silently preferred over a local one
  (must pick nearest); empty file.
