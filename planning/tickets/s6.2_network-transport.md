# mpremote: network device transport — WiFi device, handshake-discovered endpoint, TCP attach

Phase: EPIC-6 / STORY-6.2
Depends on: s5.4 (handshake plumbing), s4.1 (debugpy module on device); source delivery via mount (D2) or sync fallback (s4.2/s4.3)
Written: 2026-07-06 at HEAD 59fab8450a (micropython 18c76f7c56, micropython-lib 0d42fcd5bc)

## Context

This is the **primary v1 device path** (D3 + Q3 DECIDED: single-UART boards — which
includes ESP32_GENERIC — are network-only in v1; serial DAP, s6.1, covers only the
multi-CDC subset). The device joins WiFi, the boot script reports its own DHCP
address and probed capabilities in the `MPDBG-READY {json}` handshake over the
serial/raw-REPL **control plane**, and the tooling attaches over TCP (**data
plane**). Nobody types an IP — the "no IP literals" rule exists because the old
wrapper had nine launch configs with hardcoded addresses (`192.168.137.117`,
`my-esp32.local`).

## Scope

In scope:
- `kind=network` target flow in `do_debug`: upload/exec the boot script over
  raw-REPL, parse MPDBG-READY from the raw-REPL output stream, attach a DAP client
  (or hand the endpoint to VS Code via s7.1) to the reported `host:port`.
- WiFi credential/join handling policy for the boot flow (see Open questions).
- Reuse of the paused-device coexistence result: mount (`/remote`) may serve source
  while DAP runs over TCP (SPIKE-2.1: a paused debuggee issues no fs calls, so the
  two coexist).

Out of scope:
- mDNS discovery (the handshake already carries the address; mDNS is a possible
  later nicety, not v1).
- Serial DAP (s6.1), sync engine details (s4.2/s4.3), staleness guard (s4.3).

## Files and anchors

Verified at the stamped SHAs:

- **Launcher network probe** — `launcher/mpy_launch_debugpy.py`: `_detect_host()`
  probes for the `network` module and reads the STA_IF address
  (`import network; wlan = network.WLAN(network.STA_IF)` ~:45-55, falls back to
  bind-all on any error). Arg parsing takes `[target_module] [target_method] [port]`
  with port 0 = auto (:61). Exactly one `MPDBG-READY {json}` line printed at :90 with
  `host`, `port`, `caps`. This script becomes the artifact `do_debug` uploads/owns.
- **Port resolution** — `micropython-lib/python-ecosys/debugpy/debugpy/public_api.py`
  `listen()` (:12-93): resolves the actual bound port via `getsockname()` (:40-46);
  if the port cannot report the OS-assigned port for a port-0 request, it falls back
  to DEFAULT_PORT rather than advertising 0 (:49-55). On bare-metal this fallback is
  the expected path — the handshake port is then deterministic.
- **Capabilities** — `debugpy.get_capabilities()` (public_api.py, added in EPIC-1);
  `caps` dict carries `settrace`/`save_names`/`set_local`/`f_back` (host tests assert
  booleans: `tests/test_epic1_foundations.py::test_epic1_mpdbg_ready_handshake`).
- **Control plane** — raw-REPL exec/output via `state.transport` in this repo's
  mpremote (`micropython/tools/mpremote/mpremote/transport_serial.py`); the follow
  loop that would also service mount fs-RPC is host-side (SPIKE-2.1).
- **Reconnect caveat**: automatic reconnect after a board reset exists only in
  ampremote (5b459e2d57, `mpremote/commands.py:80 do_reconnect`), not upstream
  mpremote at 18c76f7c56 — feature-detect it (same rule as s6.1).

## Design constraints

- Endpoint ALWAYS comes from the handshake, never from user config (roadmap design
  position; acceptance criterion of STORY-5.4).
- No thread on device; the DAP server stays trace-callback-driven over its TCP
  socket — this is the shape the server already has, so the device side of this
  ticket is mostly the boot script, not the server.
- Capability honesty: `caps` from the probe; the tooling surfaces (not hides) e.g.
  placeholder local names.
- The busy-poll pause loop (10 ms) competes with WiFi housekeeping on device —
  measured, not assumed, in s6.4 (risk-register item).

## Approach sketch

1. `do_debug` for `kind=network`: ensure debugpy module present (s4.1), push/exec
   the boot script via raw-REPL with target module/method + port 0, keep reading
   raw-REPL output until the single MPDBG-READY line arrives (bounded timeout, clear
   error listing what WAS printed on failure — a crashed import must not look like a
   hang).
2. Parse via the shared handshake parser (s5.4 — one parser for subprocess stdout,
   raw-REPL, serial).
3. Attach over TCP to `host:port` from the payload. Keep the raw-REPL channel open
   as the control plane (program output, mount fs-RPC service, Ctrl-C escape hatch).
4. WiFi join: v1 assumes the device's `boot.py`/stored credentials bring the
   interface up; the boot script only *reports* the address it finds. Provisioning
   WiFi is explicitly not this command's job (document it).
5. ESP32_GENERIC is the lead target (single-UART ⇒ this is its only path);
   rpi_pico_w validates WiFi+TCP on a second port family.

## Acceptance criteria and tests

- [ ] `mpremote debug <network-target>` on ESP32_GENERIC reaches a breakpoint over
  TCP with zero IP/port literals anywhere in repo config or command line.
- [ ] Device address is taken from the handshake; deleting any cached endpoint state
  does not break the flow (re-derived every run).
- [ ] A device whose WLAN is down produces a clear diagnostic (the launcher's
  fallback binds all interfaces; the tooling must detect "no routable address
  reported" and say so) — adversarial case.
- [ ] MPDBG-READY never arrives (import crash on device) → bounded-time failure with
  the device's actual output shown, not a hang — adversarial case.
- [ ] Program `print()` output before/around the handshake line does not corrupt
  parsing (exactly-one-line contract already asserted by the EPIC-1 host test —
  extend to the raw-REPL control plane).

## Open questions

- WiFi provisioning UX when the device has no stored credentials: fail with
  instructions (recommended for v1) vs an interactive `mpremote wlan` helper —
  decide at phase entry; do not silently embed credentials in the boot script.
- Whether the control-plane raw-REPL connection stays attached for the whole session
  (live program output in the terminal) or detaches after handshake — recommend
  stay-attached; it is also what services mount fs-RPC (SPIKE-2.1).
- (From the s5.x anchor work:) `public_api.listen()` falls back to the DEFAULT port when
  a runtime cannot report an OS-assigned port after a port-0 request — on such ports the
  handshake port can differ from the actually-bound port. Verify the target port's
  `getsockname` behaviour during implementation; if unreliable, pass an explicit port
  instead of 0 on device targets.
