# mpremote+debugpy: Hot-reload iteration loop via sys.modules eviction + re-import

Phase: EPIC-4 / STORY-4.5
Depends on: STORY-2.1 (DONE — D2), s4.1 (debugpy present on device); interacts with s5.3/s5.4 session lifecycle
Written: 2026-07-06 at HEAD 59fab8450a (micropython 18c76f7c56, micropython-lib 0d42fcd5bc; ampremote/micropython 5b459e2d57)

## Context

Surfaced by SPIKE-2.1 ("Follow-up stories", `../SPIKE-2.1-mount.md`): with source
mounted from the host, the only thing standing between "edit a file" and "debug the new
code" is MicroPython's import cache — evict the target module(s) from `sys.modules` and
re-import, and the mounted edit takes effect with NO upload and NO soft-reset. This
makes mount + re-import the mainline device iteration loop (D2's corrected framing).
The one hard limit is ordinary import-caching semantics: an already-paused frame runs
already-compiled bytecode; edits apply from the next (re-)import, never inside the
current frame.

## Scope

In scope: a restart affordance in the debug flow — on session restart (DAP `restart`
request and/or a `do_debug` loop mode), evict the target module and its project-local
imports from `sys.modules`, then re-import and re-run the target, keeping the DAP
listener and mount alive.
Out of scope: live-patching the currently paused frame (impossible — set expectations
in UX text); the Debug Console monkey-patching path (s6.5); mount setup itself
(existing `mount_local`).

## Files and anchors

Verified at the stamped SHAs:

- Launcher run-target path: `launcher/mpy_launch_debugpy.py:93` —
  `target = __import__(target_module, None, None, ("*",))`, then `:109`
  `getattr(target, target_method)` and the call. The restart loop wraps exactly this:
  evict → `__import__` again → call again. The `MPDBG-READY` handshake (`:90`) must be
  emitted ONCE per process, not per iteration.
- Eviction semantics: `del sys.modules[name]` then re-import re-reads source (confirmed
  workable in the SPIKE-2.1 analysis; MicroPython honours it the same as CPython for
  filesystem imports, including from a mounted `/remote`).
- Mount mechanics (host side): `micropython/tools/mpremote/mpremote/
  transport_serial.py:319` `mount_local`, `:417` `umount_local` — the mount survives
  across re-imports because the host follow-loop keeps servicing fs-RPC (the core
  SPIKE-2.1 finding).
- DAP restart: the debugpy server (`debugpy/server/debug_session.py`) does not
  currently advertise `supportsRestartRequest`; grep confirms no `restart` handler.
  Adding one is the clean trigger; a command-side "watch + restart" loop is the
  alternative that needs no server change.

## Design constraints

- Breakpoints must survive a restart: the DAP client re-sends setBreakpoints on a new
  session, but if we keep ONE session alive across re-imports, the server's existing
  breakpoint table persists — prefer that (cheaper on-device, no re-attach latency).
- Eviction set: the target module plus modules whose `__file__` sits under the project
  source root(s) (from s5.2's target model). Never evict `debugpy.*` or stdlib —
  evicting the running debugger is self-destruction.
- The trace function must be re-installed (or confirmed still installed) after
  re-import; `debug_this_thread()` is idempotent enough today but verify.
- No soft-reset: `machine.soft_reset()` would drop the mount and the socket; the whole
  point is avoiding it.

## Approach sketch

1. Launcher grows a loop mode: run target → on completion or on an explicit restart
   command (DAP `restart` request, or a sentinel over the control channel), evict the
   project module set from `sys.modules`, re-import, re-run. Emit a per-iteration
   marker line (NOT another MPDBG-READY) so the host knows a restart happened.
2. Server side (preferred trigger): handle the DAP `restart` request — respond, unwind
   out of the paused poll loop cleanly (set continue + a restart flag the launcher
   checks), and let the launcher loop do the eviction. Advertise
   `supportsRestartRequest: true` in the initialize response only when launched in loop
   mode.
3. Command side (s5.x): `mpremote debug --loop` keeps the session attached across
   restarts; document that VS Code's restart button now "just works" on mounted
   targets.

## Acceptance criteria and tests

- Unix flow: edit a source file after the first run completes, trigger restart, hit a
  breakpoint on a line that only exists in the edited file — proves fresh source ran.
- Breakpoints set before the restart still bind after it (same session kept alive).
- `debugpy` module itself is never evicted (assert eviction set contents in a unit
  test).
- A paused session + edit + restart: the paused frame's code is unchanged until
  restart, then the new code runs — matches the documented semantics.
- Adversarial: target module imports a submodule that ALSO changed — both are evicted
  and both re-read (eviction-set correctness, the classic hot-reload hole).
