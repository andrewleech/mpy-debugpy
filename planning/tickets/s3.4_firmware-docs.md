# docs: firmware provenance + capability matrix, fetch vs build

Phase: EPIC-3 / STORY-3.4
Depends on: s3.2 (Release URLs to document), s3.3 (final variant set to document)
Written: 2026-07-06 at HEAD 59fab8450a (micropython 18c76f7c56, micropython-lib 0d42fcd5bc)

## Context
Firmware selection was the single most opaque part of the legacy workflow: four
prebuilt binaries with drifted provenance, names that promised unimplemented
features, and a "settrace by default" claim that was false on the old tree. The
docs story exists to make the firmware story *honest*: what each build is, where
it came from, what it can actually do, and how to get or reproduce it.
Constraining decisions: D4 (probe is truth), D5/Q2 ([[20260706_recomposition.md]],
CI on this repo, board set unix / rpi_pico_w / PYBD_SF6 / ESP32_GENERIC).

## Scope
In scope: one firmware guide (README section or `docs/firmware.md`) containing the
variant→flags→capabilities table, fetch instructions (`launcher/firmware.py fetch`)
and reproducible-build instructions (CI parity + local docker command), plus the
capability truth-table (real names vs `local_N`, no local editing, PERSIST off).
Out of scope: the end-user debugging guide across transports (STORY-8.4 — that doc
links here rather than duplicating); extension UI capability display (s7.2).

## Files and anchors
- `README.md` — current quick-start; the firmware guide either extends it or is
  linked from it (single entry point rule — don't create two competing quickstarts).
- `firmware/firmware.toml` header comment — already states the provenance/probe
  doctrine; the guide is its prose form and must not contradict it.
- `Makefile:9-16` — DEBUG_CFLAGS comment block documents why unix needs no extra
  flags and names the real macros; the guide supersedes/references this.
- `micropython/ports/unix/variants/standard/mpconfigvariant.h:30-42` — the
  source-of-truth for "unix standard has it on by default (integration branch)".
- `micropython/py/mpconfig.h:1840-1853` — defaults-off for everything else; the
  "for other ports set MICROPY_PY_SYS_SETTRACE=1 + MICROPY_PY_SYS_SETTRACE_LOCALNAMES=1"
  instruction comes from here.
- Probe vocabulary: `debugpy/public_api.py:136 get_capabilities()` keys
  (`settrace`, `save_names`, `set_local`, `f_back`) — the table's capability
  columns use exactly these keys.

## Design constraints
- **No capability claim the probe would contradict** (story acceptance criterion).
  Every "supports X" cell must be phrased as build intent verified by the probe at
  attach time.
- Real macro names only: `MICROPY_PY_SYS_SETTRACE_LOCALNAMES`; explicitly note
  that `_SAVE_NAMES` (old lineage docs/PRs) is a no-op on the current branch —
  readers WILL arrive from the old PRs with that spelling.
- Document that local variable *editing* is unsupported everywhere (no
  `_set_local` in any branch, D4) and that locals show as `local_N` placeholders
  when LOCALNAMES is off.
- `MICROPY_PY_SYS_SETTRACE_LOCALNAMES_PERSIST` must be documented as
  intentionally off **with the reason** (corrupts line numbers for all compiled
  code — [[20260706_recomposition.md]]), so nobody helpfully re-enables it.
- Conservative, factual tone; no "comprehensive" adjectives (user CLAUDE.md).

## Approach sketch
Structure: (1) "which firmware do I need" — one debug build per board, table of
board / artifact / flags / capabilities-as-probed; (2) "get it" — fetch + verify
commands; (3) "build it yourself" — the exact CI-equivalent local commands per
port, docker convention included; (4) "capability truth" — probe keys, what each
means in the debugger UX (placeholders, read-only locals), PERSIST warning.
Write it after s3.2/s3.3 settle the ids and URLs; drafting the capability-truth
section (4) earlier is safe — it is decision-stable now.

## Acceptance criteria and tests
- [ ] table covers all four v1 boards with flags and probe-keyed capabilities.
- [ ] fetch AND local reproducible build are both documented and both executed
  once as written (doc-test: follow the doc verbatim on a clean checkout).
- [ ] no claim the probe contradicts: cross-check every capability cell against an
  actual MPDBG-READY caps dict from the unix build (haiku agent can automate the
  unix row; device rows checked when hardware is available, noted otherwise).
- [ ] `_SAVE_NAMES` no-op warning and PERSIST rationale present.
