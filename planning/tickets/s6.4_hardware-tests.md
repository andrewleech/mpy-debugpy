# CI/manual: hardware-in-loop tests for the device debug flows

Phase: EPIC-6 / STORY-6.4
Depends on: s6.1 (serial transport), s6.2 (network transport), s4.3 (staleness guard) as applicable
Written: 2026-07-06 at HEAD 59fab8450a (micropython 18c76f7c56, micropython-lib 0d42fcd5bc)

## Context

The host-side harness (`tests/`, fake VS Code client) proves protocol behaviour
against the unix port only. The device transports carry the real risk (WiFi timing,
raw-REPL interleaving, USB CDC behaviour, the busy-poll pause loop competing with
WiFi housekeeping — an explicit risk-register item). Per the ampremote policy,
hardware validation gates upstreaming (STORY-8.3 retargets the fork PR to
`micropython:master` only after this passes).

## Scope

In scope: an on-demand (not per-commit) hardware-in-loop suite driving the full
`mpremote debug` flow on real boards; a results record (board, firmware SHA, port,
date, pass/fail per scenario) written back to `planning/` as `YYYYMMDD_hil_<board>.md`.
Out of scope: a hosted board farm; QEMU coverage (that lives in s5.5 as the CI
approximation).

## Targets

v1 board set (Q2 DECIDED): **rpi_pico_w** (native USB — serial-DAP candidate + WiFi),
**PYBD_SF6** (STM32 multi-CDC + WiFi), **ESP32_GENERIC** (single-UART ⇒ network-only
per Q3). Firmware = the s3.2 CI artifacts for the pinned integration SHA.

Boards MUST be referenced by stable `/dev/serial/by-id/<name>` paths, never raw
`/dev/ttyACMx`/`ttyUSBx` — device nodes renumber at any time (user's standing rule).

## Files and anchors

- Host harness fixtures to extend: `tests/fixtures/debuggee.py` (spawns the unix
  binary; the device variant instead drives mpremote), `tests/dap.py`,
  `tests/helpers.py` (`set_breakpoints`, `wait_for_msg`) at the stamped HEAD.
- Behavioural assertions to reuse verbatim: `tests/test_epic1_foundations.py` —
  breakpoint-stops-target, exactly-one-MPDBG-READY, readOnly locals hints. Remember
  the line-event semantics: break AFTER an assignment to observe the variable
  (D5-era finding; test comment at `tests/test_epic1_foundations.py:104-108`).
- `launcher/firmware.py` `verify`/`fetch` for artifact hashes before flashing.

## Design constraints

- Every scenario records probed `caps` alongside results — a capability mismatch
  against the manifest claim is a hard failure (STORY-3.3 rule), which is itself a
  valuable test.
- Flashing is part of the suite setup (esptool for ESP32, mboot/dfu for PYBD, UF2
  for pico_w) and must verify the artifact sha256 against `firmware/firmware.toml`
  first.
- Tests must be runnable by one command with the board plugged in; skip (not fail)
  cleanly for absent boards.

## Approach sketch

Pytest marks per board (`-m pico_w` etc.), parametrised over transport where a board
supports both. Scenario set per board/transport: flash+verify → `mpremote debug`
attach → breakpoint stop → step over/in/out → locals/globals inspect (readOnly hints
per caps) → global setVariable mutates program → continue to completion → staleness
guard trips on a deliberately stale file (where the sync path, not mount, delivers
source) → board reset mid-session behaviour (reconnect where available). Results
appended to the dated planning report.

## Acceptance criteria and tests

- [ ] ESP32_GENERIC: network flow green end-to-end.
- [ ] rpi_pico_w: network flow green; serial-DAP flow green if s6.1 landed for it.
- [ ] PYBD_SF6: at least one transport green; multi-CDC probe result recorded.
- [ ] Pause-loop impact measured on one WiFi board (ping/WLAN liveness while paused
  60 s) and recorded — closes the risk-register measurement item.
- [ ] Each run writes `planning/YYYYMMDD_hil_<board>.md` with board, by-id path,
  firmware SHA + hash-verify result, probed caps, per-scenario pass/fail.

## Workflow shape

Largely agent-driven but hardware-serialised: haiku writes the suite scaffolding,
sonnet the flash/transport glue; execution on real boards is manual-in-the-loop
(the user plugs boards; an agent runs and records). Opus reviews the results report
against the acceptance list — adversarial brief: "which claimed-green scenario
lacks evidence in the log?".

## Open questions
