## Context

The integration branches were hand-composed (merge + cherry-pick; decision record
`../20260706_recomposition.md`, D5). `mbm.toml` records the composition for
provenance only — `mbm rebase` cannot rebuild it, so every upstream-master bump is
manual archaeology. Q5 DECIDED (2026-07-06): wire reproducibility NOW.
`../../CLAUDE.md` ("Working with the integration") and `~/ampremote/` are the
reference pattern; ampremote's `mbm.toml` + `CLAUDE.md` document the known-working
flow including conflict handling.

## Scope

In scope: express both submodules' compositions as real mbm branch entries such that
`mbm rebase -s micropython` and `mbm rebase -s micropython-lib` rebuild branches that
compile and pass `make test`; wire `make integrate` to run it.
Out of scope: pushing the firmware fixes to canonical branches (s8.6, a prerequisite);
adding the future `debug` command branch (s8.1_mbm-debug-branch.md).

## Design constraints

- D5: canonical lineage is andrewleech branches composed onto current upstream master.
- Do NOT run `mbm rebase` before the branch entries are correct (risk register:
  hand-composed branches would be clobbered). Backup branches
  `mpy-debugpy-vendored-backup` exist in both submodules; create a
  `mpy-debugpy-pre-mbm` backup of the current tips before the first rebase attempt.
- `make bootstrap` must stay checkout-only; `make integrate` is the only mbm entry.
- Resulting branches must be force-pushed to the same fork refs
  (`andrewleech/micropython` @ `mpy-debugpy`, `andrewleech/micropython-lib` @
  `mpy-debugpy`) so `.gitmodules` consumers are unaffected.

## Approach sketch

The mapping hazard: mbm composes by merging named branches in order; two of our
pieces are not clean branches today.

1. micropython — create a clean locals branch first: rebase the 4 locals commits +
   2 fixes (from s8.6; keep the `.exp` newline fixup squashed into the tests commit
   or carried as-is) onto `andrewleech/pdb_support` (or onto master with pdb_support
   merged), push as `andrewleech/local_names_implementation` (refresh) or a new
   `local_names_master`. Then `mbm.toml` entries: `pdb_support` (pr_url #8767,
   pr_number 8767) then the locals branch (fork tree URL, registered by hand as a
   local-only branch, ampremote style).
1b. micropython, mpremote primitives — the mpremote PR primitives EPIC-4/5 rely on
   (verify_hash #18436, fast encoded transfer #18785, reconnect #17322,
   socket/rfc2217 #19062, QEMU PTY #18327) exist only in ampremote's tree; this
   repo's submodule carries plain upstream mpremote. They compose by the same
   mechanism — additional `mbm add-pr` entries alongside `pdb_support`/the locals
   branch. Recommendation: do NOT add them all upfront; add each when the EPIC-4/5
   ticket that needs it starts, keeping this phase-0 composition minimal. Record
   this scoping note in `mbm.toml`'s comments.
2. micropython-lib — one entry: `add-debugpy-support` (pr_url #1022). Decide whether
   Jos's 3 commits + foundations become part of a refreshed `add-debugpy-support`
   (preferred if s8.2 upstreams them there) or a second local-only branch
   `mpy-debugpy-foundations` layered after it. Until s8.2 lands, the second-branch
   form matches reality.
3. Enable `git rerere` in both submodule gitdirs; run `mbm rebase -s <sub>` for each;
   resolve conflicts once; verify `git diff <old-tip> <new-tip>` is empty or
   explainable; build + `make test`; force-push; update `mbm.toml` comments to drop
   the "not reproducible" caveat and update `../../CLAUDE.md` + ROADMAP status.
4. Point `make integrate` at `mbm rebase` for both submodules.
