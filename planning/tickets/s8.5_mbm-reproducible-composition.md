## Context

The integration branches were hand-composed (merge + cherry-pick; decision record
`../20260706_recomposition.md`, D5). `mbm.toml` records the composition for
provenance only — `mbm rebase` cannot rebuild it, so every upstream-master bump is
manual archaeology. Q5 DECIDED (2026-07-06): wire reproducibility NOW.
`../../CLAUDE.md` ("Working with the integration") and `~/ampremote/` are the
reference pattern; ampremote's `mbm.toml` + `CLAUDE.md` document the known-working
flow including conflict handling.

## Scope

In scope: express both submodules' compositions as real mbm branch entries such that
`mbm rebase -s micropython` and `mbm rebase -s micropython-lib` rebuild branches that
compile and pass `make test`; wire `make integrate` to run it.
Out of scope: pushing the firmware fixes to canonical branches (s8.6, a prerequisite);
adding the future `debug` command branch (s8.1_mbm-debug-branch.md).

- mbm 2.0.2 semantics (pinned 2026-07-14 from the installed package source,
  `micropython_branch_manager/` under the uv tool venv):
  - A remote literally named `upstream` is hardcoded: PR-by-number fetches
    (`rebase.py:178`, `pr.py:115`), the default `--target upstream/master`, and
    `sync`'s log range (`cli.py:243`). Both submodules need an `upstream` remote
    added (same URL as `origin`; keep `origin`/`andrewleech` as-is). Without it,
    rebase/add-pr/sync raise GitError.
  - `--target` is passed verbatim to git — any ref or raw SHA works. A
    pinned-target run (micropython `13303f8d2`, micropython-lib `68f8fe4`)
    rebuilds onto the SAME base as the hand-composition, giving a clean
    tree-equivalence proof before any master bump.
  - Branch entries: with `pr_number`, mbm fetches `pull/N/head` from `upstream`
    (ignores `pr_url` for content). Without `pr_number`, a fork *tree* URL in
    `pr_url` is NOT matched by the PR-URL regex (`git.py:260`), so mbm falls back
    to the pre-existing LOCAL branch of that `name`, no fetch, no push — exactly
    right for local-only branches; ensure the local branch exists before rebase.
  - Each entry is rebased independently onto the target (`git rebase
    --rebase-merges`, auto-continue via rerere) from its merge-base — a stacked
    branch (locals branch carrying pdb_support ancestry) replays its full chain;
    the double-replayed base content merges cleanly into the update branch
    because both sides are identical.
  - `rebase` NEVER touches the integration branch: the rebuild lands on
    `<integration>_update`; force-moving `<integration>` + pushing is manual.
    It DOES `git branch -f <name> rebase-<name>` for each entry (local feature
    branches move to their rebased versions).
  - Push routing is unsafe for us: feature-branch pushes go to the remote whose
    URL matches `pr_url`'s owner/repo — for upstream-PR entries that is the
    UPSTREAM repo, not the fork (always `--force`; permission failure is a
    warning). Update-branch pushes only target a remote whose URL contains
    `gitlab` (none here). Therefore ALWAYS run `mbm rebase --local` (skips all
    pushes AND the pre-fetch — run `git fetch --all` manually first) and do the
    verified force-pushes to the fork by hand.
  - Before rebuilding, mbm auto-trains rerere by replaying the existing
    integration branch's merges in a temp worktree (`rerere.py`), so the
    hand-composition's conflict resolutions are learned automatically.
  - `mbm.toml` is parsed with pydantic, extra keys ignored; inline-array and
    `[[submodules.branches]]` forms are equivalent (use the table form to match
    ampremote). Accepted branch keys: `name`, `pr_url`, `pr_number`, `title` —
    there is no `base`/`ref`/`remote` key.

## Design constraints

- D5: canonical lineage is andrewleech branches composed onto current upstream master.
- Do NOT run `mbm rebase` before the branch entries are correct (risk register:
  hand-composed branches would be clobbered). Backup branches
  `mpy-debugpy-vendored-backup` exist in both submodules; create a
  `mpy-debugpy-pre-mbm` backup of the current tips before the first rebase attempt.
- `make bootstrap` must stay checkout-only; `make integrate` is the only mbm entry.
- Resulting branches must be force-pushed to the same fork refs
  (`andrewleech/micropython` @ `mpy-debugpy`, `andrewleech/micropython-lib` @
  `mpy-debugpy`) so `.gitmodules` consumers are unaffected.

## Approach sketch

The mapping hazard: mbm composes by merging named branches in order; two of our
pieces are not clean branches today.

1. micropython — create a clean locals branch first: rebase the 4 locals commits +
   2 fixes (from s8.6; keep the `.exp` newline fixup squashed into the tests commit
   or carried as-is) onto `andrewleech/pdb_support` (or onto master with pdb_support
   merged), push as `andrewleech/local_names_implementation` (refresh) or a new
   `local_names_master`. Then `mbm.toml` entries: `pdb_support` (pr_url #8767,
   pr_number 8767) then the locals branch (fork tree URL, registered by hand as a
   local-only branch, ampremote style).
1b. micropython, mpremote primitives — the mpremote PR primitives EPIC-4/5 rely on
   (verify_hash #18436, fast encoded transfer #18785, reconnect #17322,
   socket/rfc2217 #19062, QEMU PTY #18327) exist only in ampremote's tree; this
   repo's submodule carries plain upstream mpremote. They compose by the same
   mechanism — additional `mbm add-pr` entries alongside `pdb_support`/the locals
   branch. Recommendation: do NOT add them all upfront; add each when the EPIC-4/5
   ticket that needs it starts, keeping this phase-0 composition minimal. Record
   this scoping note in `mbm.toml`'s comments.
2. micropython-lib — one entry: `add-debugpy-support` (pr_url #1022). Decide whether
   Jos's 3 commits + foundations become part of a refreshed `add-debugpy-support`
   (preferred if s8.2 upstreams them there) or a second local-only branch
   `mpy-debugpy-foundations` layered after it. Until s8.2 lands, the second-branch
   form matches reality.
3. Add `upstream` remotes; `git fetch --all`; back up tips; then per submodule run
   `mbm rebase -s <sub> --local --target <pinned base SHA>` (micropython
   `13303f8d2`, micropython-lib `68f8fe4`) so the rebuild lands on
   `mpy-debugpy_update` with no pushes; resolve conflicts once (rerere is
   auto-trained from the existing merges); verify `git diff <hand-composed tip>
   <update tip>` is empty apart from s8.6's new test; build + `make test`;
   force-move `mpy-debugpy` to the update tip and force-push fork refs by hand;
   update `mbm.toml` comments to drop the "not reproducible" caveat and update
   `../../CLAUDE.md` + ROADMAP status. (Default-target runs against latest
   master become routine afterwards; not part of this phase.)
4. Point `make integrate` at `mbm rebase` for both submodules.

## Execution outcome (2026-07-15)

Both compositions rebuilt via `mbm rebase --local` against pinned targets
(micropython `13303f8d2`, micropython-lib `68f8fe4`) with ZERO conflicts in
either submodule; the branches replayed cleanly, rerere never fired. Composed
tips force-pushed to the fork: micropython `mpy-debugpy` `18c76f7c5` ->
`6863e938a1`, micropython-lib `mpy-debugpy` `0d42fcd` -> `d7b297dfd1`.
Executed as a multi-agent workflow (prepare -> parallel rebases -> test
battery -> opus review -> publish); details in
`../20260715_phase0-canonical-branches-mbm.md`.

- `upstream` remotes added to both submodules (mbm hardcodes the name);
  `mpy-debugpy-pre-mbm` backups sit at the old tips; ssh push URLs set on the
  `andrewleech` remotes.
- `mpy-debugpy-foundations` created at `e359a95` and pushed to the fork; it
  and `local_names_implementation` are the two local-only entries (fork
  `/tree/` pr_url, no pr_number). `pdb_support` (#8767) and
  `add-debugpy-support` (#1022) are PR-number entries fetched live from
  upstream at each rebase.
- mbm's `branch -f` moves local feature branches to their rebased-onto-target
  versions as a side effect; those were reset back to the canonical fork tips
  afterwards (`local_names_implementation` @ `7ae5f769c`, `pdb_support` @
  `3280c2a5e`, `mpy-debugpy-foundations` @ `e359a95`) so the fork remains the
  single source of truth. Expect this after every rebase run.
- `make integrate` wired: fetch both submodules, `mbm rebase --local` both,
  then echo the manual verify/force-move/force-push steps. Never run
  `mbm rebase` without `--local` (its push routing targets upstream).
- Review note to carry: a default-target (`upstream/master`) run is now
  routine but moves every pin; do the first one deliberately, together with
  the firmware work (s3.2), not as a side effect.
