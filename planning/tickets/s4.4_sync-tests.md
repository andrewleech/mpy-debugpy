## Context

EPIC-4's whole value is "staleness impossible without a warning" — that is only true if
the no-op/diff/verify-fail/stale paths are pinned by tests that run without hardware.
The existing host-side DAP harness gives the pattern to extend; QEMU PTY detection
(#18327, ampremote lineage) lets device-transport paths run in CI.

## Scope

In scope: automated tests for s4.1 (install caching), s4.2 (incremental sync), s4.3
(staleness guard + pathMappings generation), runnable in CI without physical hardware.
Out of scope: hardware-in-loop validation (s6.4); DAP protocol behaviour tests (already
covered by `tests/test_epic1_foundations.py` and `tests/test_dbg_*.py`).

## Design constraints

- Two test tiers: (a) fast unit tests with a FAKE transport (spy on
  `fs_writefile`/exec calls — no firmware needed) proving no-op/diff/delete/manifest
  logic; (b) integration tests against unix (and QEMU PTY where the transport layer is
  exercised) proving end-to-end behaviour.
- Transport-call assertions must count actual write calls, not parse log text.
- No hardcoded ports, paths, or device names; serial devices in any optional
  hardware-marked tests use `/dev/serial/by-id/...` and are skipped when absent.

## Approach sketch

1. Build a `FakeTransport` recording `fs_writefile`/`fs_readfile`/exec calls, with an
   in-memory device filesystem — the unit under test is the sync/install logic, which
   per s4.1/s4.2 must be transport-agnostic functions.
2. Unit tests: fresh install; cached no-op; single-file change; local delete; mpy-cross
   version bump → full reinstall; verify-fail → manifest untouched; interrupted sync →
   self-heal.
3. Integration: unix flow — sync to a temp dir "device" (or QEMU PTY when wired),
   deliberately stale one file, assert the s4.3 warning text and (strict mode) refused
   attach through the real `fake_vscode` client.
4. pathMappings property test: generated mappings are absolute; round-trip identity
   through `pdb_adapter._filename_as_debugee`/`_filename_as_debugger`.
5. Mark QEMU-dependent tests with a skip-if-unavailable guard so CI degrades cleanly.

## Acceptance criteria and tests

- All s4.1/s4.2/s4.3 acceptance criteria have at least one test each (the adversarial
  ones included: interrupted install, lying manifest).
- Unit tier runs with no firmware binary present.
- Full suite green in CI via `make test`; QEMU tier green where QEMU is installed,
  skipped (not failed) where not.

## Workflow shape

Test authoring on haiku inside the EPIC-4 workflow, with an opus review pass checking
the tests exercise behaviour (transport-call counts, real DAP sessions) rather than
implementation details; loops with the s4.1–s4.3 implementers.
