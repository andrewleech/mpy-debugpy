# mpremote: one handshake parser across all control planes (control/data-plane split)

Phase: EPIC-5 / STORY-5.4
Depends on: s5.1 (skeleton), s5.3 (unix flow — first consumer); D3/Q3 (serial DAP deferred)
Written: 2026-07-06 at HEAD 59fab8450a (micropython 18c76f7c56, micropython-lib 0d42fcd5bc; ampremote/micropython 5b459e2d57)

## Context

Every transport launches the same boot script and gets back the same
`MPDBG-READY {json}` line, but over a different byte stream: subprocess stdout
(unix), raw-REPL exec output (serial device), or serial output from a device that is
about to be reachable over the network (network device before/after it has an IP).
The design position mandates the split: the CONTROL plane carries the handshake, the
DATA plane carries DAP. Per D3/Q3 (2026-07-06) the v1 data plane is TCP only; the
serial DAP tunnel is deferred but the parser and endpoint types must not preclude it
(SPIKE-2.2 retains the framing design for later).

## Scope

In scope: a single handshake reader parameterised over a "line source" (unix
subprocess pipe; raw-REPL output stream via mpremote transport), returning a
validated endpoint+caps record; endpoint resolution rules (see constraints);
timeout/error semantics shared by all callers.

Out of scope: the transports themselves (s5.3 unix, s6.1/s6.2 device flows consume
this); the serial DAP tunnel data plane (deferred by Q3); mDNS or any discovery
other than the handshake.

## Design constraints

- **Endpoint always comes from the device/subprocess, never from user config.**
- **Never emit a raw `0.0.0.0` endpoint.** Resolution rules by control plane:
  unix subprocess → `127.0.0.1`; serial/network device reporting `0.0.0.0` → the
  transport's known address if any, else a hard error telling the user the device
  has no network address (do not guess). A device reporting a real address is taken
  verbatim.
- Exactly-one-line contract: duplicate or zero `MPDBG-READY` lines within the
  timeout is an error (matches the emit-side guarantee and the existing test).
- Design for a future non-TCP data plane: the parsed record should carry an
  endpoint kind (tcp host/port now; a serial-channel descriptor later per
  SPIKE-2.2) rather than a bare `(host, port)` tuple baked through every caller.
- Human banner lines around the handshake (`Debugpy listening on ...`,
  `public_api.py:57`) are noise — never parsed, never depended on.

## Approach sketch

`_read_mpdbg_ready(read_chunk, timeout) -> DebugEndpoint` where `read_chunk` is a
callable returning available bytes/str ('' when none): the unix caller passes a
non-blocking-pipe reader, the serial caller passes a transport-backed reader.
Accumulate, scan for `MPDBG-READY ` at line start, `json.loads` the rest of that
line, validate schema (host str, port int, caps dict of bools), then apply the
endpoint-resolution rules above with the caller-supplied context (control-plane
kind + known device address). Return record: `{kind: "tcp", host, port, caps,
raw_host}` keeping the reported `raw_host` for diagnostics.

## Acceptance criteria and tests

- [ ] one parser implementation serves unix stdout and raw-REPL serial output
  (unit-test with synthetic chunked streams; integration via s5.3/s6.x flows).
- [ ] `host: "0.0.0.0"` from a unix subprocess resolves to `127.0.0.1`; from a
  device with no known address it is a hard, explanatory error; a real device
  address passes through verbatim. No code path ever hands `0.0.0.0` to a DAP
  client.
- [ ] zero lines (timeout) and two lines both error, with the captured stream tail
  in the message.
- [ ] schema violations (port not int, caps not dict, non-bool cap) error rather
  than propagate garbage.
- Adversarial: handshake split across chunk boundaries mid-line; `MPDBG-READY`
  appearing as a substring inside another line (must match at line start only);
  trailing REPL echo/prompt bytes after the JSON on serial.
